# skill-definition

## 目的

新しい skill を、実際に再利用できる粒度で定義する。

この skill は「人格」を作るためのものではない。
「繰り返し使う作業単位」を定義するためのものとして扱う。

## 基本原則

- 1 skill = 1 job を原則とする
- 抽象的な肩書きではなく、具体的な作業名にする
- skill の責務と、毎回の個別指示を分ける
- 恒常的なプロジェクト方針は skill ではなく AGENTS.md に置く
- 同じ内容を毎回書く必要があるなら skill 化を検討する
- まだ不安定な運用は、まず軽い指示として試し、繰り返し使われることを確認してから skill 化する

## まず確認すること

新しい skill を提案・定義するときは、最初に次を整理する。

1. この skill は何の作業を担当するか
2. いつ使うか
3. いつ使わないか
4. 何を入力として想定するか
5. どのような出力を期待するか
6. AGENTS.md に書くべき内容ではないか
7. 既存 skill と重複していないか

## 命名ルール

以下を優先する。

- 良い例
  - vue-component-design
  - vue-code-review
  - ui-layout-design
  - youtube-player-integration

- 避けたい例
  - vue-expert
  - senior-reviewer
  - design-master

名前は「肩書き」ではなく「仕事」を表すものにする。

## 設計手順

新しい skill を作るときは、次の順で設計する。

### 1. job を一文で定義する

例:

- Vue コンポーネント分割と state 配置を設計する
- UI レイアウト方針を決める
- 既存 skill.md の改善点をレビューする

### 2. 境界を定義する

必ず以下を書く。

- 使う場面
- 使わない場面

### 3. 毎回変わる条件を分離する

SKILL.md には固定ルールだけを書く。
今回だけの要件、画面固有の制約、数値条件は、通常のプロンプトで追加する前提にする。

### 4. 出力形式を定める

出力形式はなるべく固定する。
例:

- 問題点
- 改善案
- 修正版
- 実装方針
- コピペ用テンプレート

### 5. 最小構成で始める

最初から詳細にしすぎない。
2〜3回同じズレが出たら追記する。

## 出力ルール

この skill を使って新しい skill を定義するときは、原則として次を返す。

1. skill 名
2. この skill の目的
3. 使う場面
4. 使わない場面
5. 入力例
6. 出力例
7. `SKILL.md` の草案
8. AGENTS.md に書くべき内容があればその指摘
9. 既存 skill と重複する場合は統合案

## yz-player 向け観点

yz-player のような軽量な単一画面 Vue アプリでは、次を特に重視する。

- 過剰な抽象化を避ける
- 1画面完結を優先する
- skill の分割を細かくしすぎない
- セトリ変換 / UI レイアウト / YouTube 連携 / GitHub Pages 公開のように、壊れやすい問題領域ごとに分ける
- CSS ライブラリ名よりも、作業内容を中心に定義する
- AGENTS.md の主要利用体験・データ互換ルールと矛盾しないか確認する

## 返答スタイル

- まず「その skill は作る価値があるか」を判断する
- 不要なら skill 化しない案も出す
- 必要なら最小構成から提案する
- 抽象論だけで終わらず、必ずそのまま使える `SKILL.md` 草案まで出す
