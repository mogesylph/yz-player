# skill-design-review

## 目的

既存の skill を、より再利用しやすく、ブレにくく、保守しやすい形に改善する。

この skill は、skill の品質レビュー専用である。
アプリのコードレビューそのものは担当しない。

## レビュー観点

既存 skill を確認するときは、次の観点で見る。

### 1. 責務が明確か

- 1 skill = 1 job になっているか
- 複数の役割が混ざっていないか
- 「設計」「レビュー」「実装」が混在していないか

### 2. 抽象的すぎないか

- 肩書きだけで終わっていないか
- 何をする skill なのかが明確か
- 使う場面 / 使わない場面が分かるか

### 3. 毎回の指示と固定ルールが分離されているか

- 個別案件の条件が SKILL.md に入りすぎていないか
- 恒常的な原則だけが残っているか

### 4. 他と重複していないか

- 既存 skill と責務が重なっていないか
- AGENTS.md に置くべきルールを skill 側に書いていないか

### 5. 出力が再利用しやすいか

- 出力形式がある程度固定されているか
- 毎回ほしい成果物が得られるか
- 抽象論だけで終わらないか

### 6. 長すぎないか

- 説明が冗長ではないか
- 重要なことが埋もれていないか
- ルールが過剰に細かくなっていないか

## よくある問題

次の問題を見つけたら、優先して改善する。

- skill 名が抽象的
- 説明文だけでは trigger 条件が分からない
- 使わない場面が定義されていない
- 出力形式がない
- 具体例がない
- プロジェクト方針と個別タスク指示が混在している
- 同じ内容を別 skill でも繰り返している

## 改善方針

レビュー時は、次の順で改善案を考える。

1. 不要な記述を削る
2. 責務を一つに絞る
3. 名前を job ベースに直す
4. 使う場面 / 使わない場面を明記する
5. 出力形式を追加する
6. 例を最小限だけ加える

## 出力ルール

この skill を使ったときは、原則として以下の形で返す。

1. 現状の問題点
2. その skill の良い点
3. 改善方針
4. 必要なら新しい skill 名
5. 修正版 `SKILL.md`
6. AGENTS.md に移すべき内容
7. 分割または統合したほうがよい場合はその提案

## 判断基準

レビューの結果、次のいずれかを明確に判断する。

- このままでよい
- 少し修正すればよい
- 名前変更が必要
- 分割が必要
- 他 skill と統合したほうがよい
- skill にせず AGENTS.md に移したほうがよい

## yz-player 向け観点

yz-player のような軽量な単一画面 Vue アプリでは、次を優先する。

- skill を細かく分けすぎない
- セトリ変換 / UI レイアウト / YouTube 連携 / GitHub Pages 公開など、壊れやすい境界で分ける
- AGENTS.md にある恒常ルールを skill 側で重複させすぎない
- CSS ライブラリ名だけで skill を作らない
- 運用して初めて見えたズレだけを追記する
- 完璧さより再利用性を優先する

## 返答スタイル

- まず、現状の skill は本当に改善が必要かを判断する
- 改善が必要な場合は、問題点を優先度順に並べる
- 可能なら全面書き直しではなく最小修正案から出す
- 最後に、そのまま置き換え可能な `SKILL.md` を返す
