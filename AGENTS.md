# AGENTS.md

## Project Overview

`yz-player` は、YouTube 動画内の指定区間を曲単位で連続再生する軽量な単一画面アプリである。

- フロントエンド中心の Vue アプリ
- 単一画面で完結
- YouTube プレイヤー、プレイリスト、セトリ変換が主要機能
- 仕様は `docs/` 配下の各仕様ドキュメントを参照する

このプロジェクトでは、見た目の作り込みよりも「主要な利用体験を壊さず、セトリ変換を変更しやすく保つこと」を優先する。

---

## Core Principles

### 1. 主要な利用体験を尊重する

- 既存の挙動を変更する場合は、理由を明確にする
- `songlist.tsv`、localStorage、主要な操作導線を軽視しない
- UI を作り直す場合も、主要な操作導線は維持する

---

### 2. シンプル最優先

- 過剰な抽象化は禁止
- 小規模アプリとして最短で理解できる構造を優先
- 「便利そう」ではなく「今必要か」で判断する
- ただし、セトリ変換・YouTube 連携・ファイル入出力は壊れやすい境界なので、適切に分離する

---

### 3. 単一画面完結

- 画面遷移を増やさない
- グローバル状態は極力持たない
- 状態は画面内または composable に閉じる
- Pinia などのグローバルストアは、必要になってから検討する

---

### 4. セトリ変換を中核機能として扱う

セトリ変換はこのアプリのキーポイントである。

- Vue コンポーネントに正規表現や補完ロジックを閉じ込めない
- 形式追加がしやすい構造にする
- 解析できなかった行を黙って捨てない
- 曲長補完の仕様を明文化する
- 変換ロジックにはテストを用意する

---

## UI / Layout Rules

### レイアウト原則

- YouTube プレイヤーを主要 UI として扱う
- プレイヤー、操作ボタン、プレイリスト、セトリ変換入口を明確に分ける
- デザインは最低限でよい
- 後で見直せるよう、レイアウトとアプリロジックを混ぜない
- 不要な余白、装飾、カード化を増やさない

---

### UI 優先順位

UI 設計は次の順で判断する。

1. YouTube プレイヤー
2. 再生操作
3. プレイリスト操作
4. セトリ変換
5. テーマ、装飾、余白

---

### 具体ルール

- 動画の縦横比を崩さない
- 操作ボタンはコンパクトにまとめる
- プレイリストは現在曲が分かるようにする
- セトリ変換は開きやすく、閉じやすくする
- 解析エラーや空状態は最低限分かるようにする
- レイアウト調整は `gap` を優先し、margin の多用を避ける
- padding は必要最小限にする

---

### レイアウト変更時の必須確認

- 動画サイズが不必要に縮んでいないか
- プレイヤーの縦横比が崩れていないか
- UI 追加で操作領域が圧迫されていないか
- スマホ幅でも再生、次曲、セトリ追加が最低限使えるか
- 現在曲とプレイリストの関係が分かるか

---

## Vue Design Rules

### コンポーネント設計

- コンポーネントは分けすぎない
- 責務が明確に分かれる場合のみ分割する
- UI とロジックが混ざって読みづらくなったら分割する
- プレイヤー、操作ボタン、プレイリスト、セトリダイアログは分割候補とする
- 小さすぎるラッパーコンポーネントを増やさない

---

### 推奨する分割

必要に応じて次の単位を目安にする。

- `App`: アプリ全体の接続
- `AppShell`: アプリバー、テーマ、メイン枠
- `PlayerView`: プレイヤー画面の接続
- `YouTubeViewport`: YouTube 表示領域
- `PlaybackControls`: 再生、次曲、音量、再生モード
- `PlaylistList`: プレイリスト表示、選択、削除
- `SetlistDialog`: セトリ変換 UI

この構成は目安であり、不要なら無理に作らない。

---

### state 管理

- state はローカル優先
- 画面をまたがらない状態はコンポーネントまたは composable に閉じる
- localStorage のキーは定数化する
- プレイリスト、現在曲、再生状態は単純なデータ構造を維持する
- グローバルストア導入は最後の手段とする

---

### composables / services

必要な境界にだけ作る。

作ってよいもの:

- `usePlaylist`
- `usePlaybackController`
- `useYouTubePlayer`
- `useSongMaster`
- `useSetlistDraft`
- セトリ変換の pure function
- プレイリストファイル import / export service
- YouTube video ID parser

避けるもの:

- 1 回しか使わない薄すぎる composable
- 名前だけ立派で実体のない service
- 将来の汎用化を狙った抽象化

---

## Setlist Conversion Rules（最重要）

セトリ変換は Vue コンポーネントから独立した pure function として扱う。

### 処理の基本パイプライン

```text
入力
  |
  v
videoId / URL の検証
  |
  v
行分割
  |
  v
行の正規化
  |
  v
timestampPatterns による行解析
  |
  v
曲名 / アーティストの正規化
  |
  v
曲長マスタ照合
  |
  v
end 補完
  |
  v
PlaylistItem[] + ParseIssue[]
```

---

### timestamp pattern の追加ルール

- パターンには `id`, `label`, `examples`, `regex`, `parse` を持たせる
- 追加時はサンプルとテストを必ず足す
- 既存パターンの優先順位を変える場合は、既存ケースが壊れないか確認する
- 正規表現をコンポーネント内に直接書かない

---

### end 補完ルール

`end` の決定順は以下とする。

1. 曲長マスタに一致した場合: `start + length`
2. 次の曲があり、次曲まで 360 秒未満の場合: 次曲の `start`
3. 次の曲がない、または次曲まで 360 秒以上の場合: `start + 360`

この仕様を変更する場合は、ドキュメントとテストも更新する。

---

### 解析失敗の扱い

- 読み取れない行を単に捨てない
- `ParseIssue` として line number、元行、理由を残す
- UI では最低限、読み取れなかった行数が分かるようにする
- 変換成功件数が 0 の場合は、ユーザーに明確に伝える

---

## YouTube Integration Rules（重要）

YouTube iframe は通常の DOM と挙動が異なるため、以下を必ず守る。

- レイアウト確定後に初期化する
- 親要素サイズ依存であることを前提にする
- 縦横比を CSS で明示する
- 動作確認は `npm run dev` で起動した `http://localhost:<port>/` を使う
- `127.0.0.1` 起動では、動画によって iframe 再生可否が `localhost` と変わることがある
- 再描画で iframe が崩れる可能性を考慮する
- YouTube IFrame API の loader は singleton にする
- `window.onYouTubeIframeAPIReady` の上書き衝突に注意する
- Player API はコンポーネントから直接広く触らず、ラッパーに閉じ込める

## Data / Compatibility Rules

### Sample Data

ドキュメント、テスト、コード例で曲名とアーティスト名のサンプルが必要な場合は、原則として次を使う。

- 曲名: `l'aster`
- アーティスト名: `柚羽まくら`

実データの事実を説明する場合を除き、別の実名サンプルを増やさない。

---

### PlaylistItem

内部データは単純に保つ。

```ts
type PlaylistItem = {
  id: string
  videoId: string
  title: string
  artist?: string | null
  start: number
  end: number
}
```

- 内部では `title` と `artist` を分ける
- 表示やファイル出力時だけ必要な形式へ整形する
- 重複判定は原則 `videoId`, `start`, `end` を使う

---

### プレイリストファイル

- 保存形式は version を持たせる
- import は寛容にする
- import 後の内部データは厳格に正規化する

---

### `songlist.tsv`

- `public/songlist.tsv` から読み込む
- ヘッダー行を曲データとして扱わない
- 検索キーや表記ゆれ処理は `songMatcher` に閉じ込める
- 曲長マスタが読めなくても、次曲開始時刻または 360 秒補完で動くようにする

---

## Coding Guidelines

### 優先順位

1. 主要な利用体験を壊さない
2. 読みやすさ
3. シンプルさ
4. テストしやすさ
5. 拡張性

---

### 禁止事項

- 不要な抽象化
- 未来のためだけの設計
- 汎用化のしすぎ
- ライブラリ依存の過剰導入
- セトリ変換ロジックを Vue コンポーネントに埋め込むこと
- YouTube Player API をあちこちから直接操作すること
- 保存形式を過去形式や外部都合に無理に合わせること

---

### テスト方針

セトリ変換は優先してテストする。

最低限の対象:

- YouTube URL / video ID の抽出
- 主要なタイムスタンプ形式
- 曲名 / アーティスト分割
- 絵文字入り行
- タブ区切り行
- 解析失敗行
- 曲長マスタ一致時の `end`
- 曲長マスタ不一致時の `end`
- 最後の曲の `end`
- シャッフル 1 曲時の挙動

---

## Decision Rules

実装に迷った場合は以下で判断する。

### Yes の場合のみ採用

- 今すぐ必要か？
- 主要な利用体験の維持に役立つか？
- コード量または認知負荷が減るか？
- テストしやすくなるか？
- セトリ変換の追加・変更が安全になるか？

### No が 1 つでもあれば慎重にする

- 将来使うかもしれない
- 他でも使えそう
- 一般的には良い設計
- 見た目が少し良くなるだけ
- 抽象名はきれいだが処理は増える

- 同じ指示を 2 回以上書いた
- 判断がブレる
- レビュー観点を固定したい
- セトリ変換の追加手順を固定したい

---

### 使用ルール

- 新規 skill 作成時は `skill-creator` を使う
- 最初から完璧にしない
- 運用しながら改善する
- プロジェクト固有ルールはこの `AGENTS.md` と矛盾しないようにする

---

## Output Expectations（AI への指示）

### 必須

- 抽象論で終わらない
- 実装可能な形で出す
- 既存ドキュメントを確認してから提案する
- 仕様変更を含む場合は、変更点を明示する
- 必要ならコード例を出す

---

### 優先

- 最小構成で提案する
- 不要な複雑化を避ける
- 現状の構造と主要な利用体験を尊重する
- セトリ変換への影響を明示する
- YouTube プレイヤーへの影響を確認する

---

## Anti-Patterns

以下は避ける。

- Vue の過剰コンポーネント分割
- 無意味な composables 化
- CSS の過剰抽象化
- レイアウトより装飾優先
- YouTube プレイヤーを小さくする変更
- セトリ変換の正規表現を UI コンポーネントに追加する変更
- 解析失敗行を無視する実装
- 過去形式との互換を絶対条件として設計を歪める変更
- localStorage キーを散らばらせる実装

---

## Final Principle

このプロジェクトでは:

> 「正しい設計」より「主要な利用体験を壊さず、セトリ変換を育てやすい設計」を優先する
